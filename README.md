# Nash Agent

Nash Agent — локальный продуктовый агент для менеджеров продукта, который превращает описание фичи, PRD или стратегический кейс в многопользовательскую игровую модель. Агент выделяет участников, строит пространство стратегий, оценивает профили через LLM, ищет устойчивые равновесия Нэша и собирает результат в управленческий отчёт.

Сервис помогает до начала разработки понять, какие акторы могут поддержать или сломать запуск, где возникают координационные риски, какие ограничения стоит поставить и какой следующий ход выглядит рациональным для менеджера продукта.

## Что умеет сервис

- Выделяет игроков, границы игры, стратегии и допущения из текстового описания кейса.
- Строит до 64 стратегических профилей и оценивает их пакетами, чтобы длинный анализ не зависел от одного гигантского LLM-запроса.
- Показывает live progress: завершённые этапы, текущий этап, таймер анализа, число профилей и счётчик вида `4 из 16 проф.`.
- Рассчитывает равновесия Нэша, Nash score, confidence, риски, стратегические ходы и guardrails.
- Формирует развёрнутую статью агента на русском языке в стиле продуктового объяснения механики игры.
- Собирает `План действий для продакт-менеджера`: что делать дальше, почему сейчас, стратегические ходы, план проверки, ограничители запуска и сценарии контрходов.
- Визуализирует акторов, стратегии, выигрыши, попарные срезы и ветвления стратегий через Recharts.
- Хранит историю анализов, статус, длительность, число профилей и число фрагментов ответа LLM.
- Экспортирует полный анализ в печатный PDF-документ на белом фоне с минимальными рамками.
- Позволяет выбрать провайдера LLM в разделе `Настройки`: локальная LLM через LM Studio или облачная Yandex AI Studio.

## Архитектура

Проект — full-stack TypeScript-приложение.

- `client/`
  - React + Wouter.
  - Tailwind CSS и shadcn/ui.
  - TanStack Query для API-запросов.
  - Recharts для графиков и Sankey-like представлений.
  - Основные страницы:
    - `NewAnalysis.tsx` — создание кейса и live progress во время анализа.
    - `AnalysisView.tsx` — полный результат анализа, визуализации, статья агента, PM Decision Pack и ссылка на PDF.
    - `History.tsx` — список кейсов, статусы, длительность, количество профилей и фрагментов.
    - `Settings.tsx` — выбор LLM-провайдера и параметры подключения.

- `server/`
  - Express API.
  - SSE-стриминг прогресса анализа.
  - Очередь запросов к LLM, чтобы этапы выполнялись последовательно и управляемо.
  - Интеграция с OpenAI-compatible API для LM Studio и Yandex AI Studio.
  - Генерация PDF через `pdfkit`.
  - Локальный debug-режим без реальной LLM.

- `shared/`
  - Общая Drizzle-схема и TypeScript-типы для клиента и сервера.

- `SQLite + Drizzle`
  - Локальное хранение кейсов и результатов в `nash.db`.

## Поток анализа

1. Пользователь вводит название кейса, тип решения и описание ситуации.
2. Агент готовит запрос к LLM и выделяет игроков, границы игры, стратегии и допущения.
3. Бэкенд ограничивает модель до управляемого числа игроков и профилей.
4. Стратегические профили оцениваются пакетами, по умолчанию по 4 профиля за один LLM-запрос.
5. Бэкенд объединяет оценки, рассчитывает равновесия, Nash score, confidence и сводные метрики.
6. LLM генерирует развёрнутую статью агента на русском языке.
7. LLM собирает `План действий для продакт-менеджера`.
8. Результат сохраняется в истории и становится доступен как интерактивная страница и PDF-отчёт.

Такой пайплайн заменяет один длинный запрос несколькими контролируемыми фазами. Это снижает риск зависаний на поздних этапах, позволяет показывать точный прогресс и повторять проблемный запрос без потери уже рассчитанных шагов.

## LLM-провайдеры

Провайдер выбирается на странице `Настройки`.

### Локальная LLM через LM Studio

Подходит для приватной работы с чувствительными продуктовыми кейсами.

```env
LLM_PROVIDER=local
OPENAI_BASE_URL=http://127.0.0.1:1234/v1
OPENAI_API_KEY=lm-studio
LLM_MODEL=auto
LLM_TIMEOUT_MS=900000
LM_STUDIO_TTL_SECONDS=86400
```

Если LM Studio возвращает ошибку вида `Model reloaded`, анализ не считается безвозвратно упавшим: интерфейс предлагает проверить загруженную модель и повторить тот же запрос.

### Yandex AI Studio

Подходит для облачного режима, когда нужна более стабильная или производительная модель. API-ключ не хранится в README и не должен попадать в git. Пользователь вводит его на странице `Настройки` или задаёт через локальный `.env`.

```env
LLM_PROVIDER=yandex
YANDEX_AI_STUDIO_BASE_URL=https://ai.api.cloud.yandex.net/v1
YANDEX_AI_STUDIO_API_KEY=
YANDEX_AI_STUDIO_PROJECT_ID=<your-project-id>
YANDEX_AI_STUDIO_MODEL=<your-model-uri>
YANDEX_AI_STUDIO_TEMPERATURE=0.8
YANDEX_AI_STUDIO_MAX_OUTPUT_TOKENS=40000
```

Локальный файл `.nash-agent-settings.json` используется для пользовательских настроек и исключён из git.

## Русский язык в результатах

JSON от LLM валидируется на человекочитаемые поля. Если модель возвращает английский текст в полях результата, анализ завершается с понятной ошибкой и просьбой переписать текстовые значения на русском языке. Это нужно, чтобы итоговый UI и PDF не смешивали языки.

## PDF-отчёт

На странице анализа в самом низу есть ссылка на скачивание PDF:

```text
/api/analyses/:id/pdf
```

Отчёт включает все основные разделы анализа: вводные данные, состав игры, допущения, инсайты, равновесия, попарные срезы, стратегические ветвления, развёрнутую статью агента и план действий для продуктового менеджера. Документ генерируется для печати: белый фон, шрифт 10 pt, компактные рамки и без пустых страниц в конце.

## Локальный запуск

Установите зависимости:

```bash
npm install
```

Создайте `.env` на основе `.env.example`:

```bash
cp .env.example .env
```

Запустите приложение:

```bash
npm run dev
```

По умолчанию сервис поднимается на:

```text
http://127.0.0.1:5000/#/
```

Порт можно изменить через `PORT`. В локальной разработке удобно использовать `PORT=5001`, если `5000` уже занят.

## Проверка

Проверка TypeScript:

```bash
npm run check
```

Проверка PDF после изменений в генераторе отчётов:

```bash
curl -L http://127.0.0.1:5001/api/analyses/39/pdf -o /tmp/nash-analysis.pdf
pdfinfo /tmp/nash-analysis.pdf
```

## Важные файлы, которые не коммитятся

- `.env` — локальные переменные окружения.
- `.nash-agent-settings.json` — пользовательские настройки LLM и сохранённые ключи.
- `nash.db` — локальная SQLite-база с анализами.

Не добавляйте API-ключи в README, `.env.example`, исходный код или историю git.
