# Nash Agent

Nash Agent — продуктовый агент для менеджеров продукта: **Game & Complexity Theory for PM**. Он превращает описание фичи, продуктовой стратегии или рыночного запуска в модель поведения участников, показывает устойчивое целевое состояние и проверяет, достижимо ли оно в реальной динамике рынка.

Сервис помогает до разработки увидеть, какие акторы поддержат запуск, где появятся координационные риски, какие ранние сигналы отслеживать и какой следующий ход рационален для продакт-менеджера.

## Режимы анализа

- `Game Theory` — находим устойчивое состояние, если игроки рациональны и правила игры заданы. Получаем целевое состояние системы.
- `Complexity Theory` — смотрим, что произойдёт, если игроки учатся, ошибаются, а система идёт по разным траекториям.
- `Совмещённый анализ` — смотрим, насколько после запуска возможно достичь устойчивого состояния в реальном рынке.

## Что умеет сервис

- Автоматически выделяет игроков, границы игры, стратегии, допущения и внешних участников из текстового описания кейса.
- Строит до 64 стратегических профилей и оценивает их пакетами, чтобы длинный анализ не зависел от одного гигантского LLM-запроса.
- Рассчитывает равновесия Нэша, Nash score, confidence, риски, выигрыши, стратегические ходы и ограничители запуска.
- В режиме Complexity Theory строит ограниченную адаптивную симуляцию: сценарии, обратные связи, пороговые переломы, режимы системы, ранние сигналы и пространство будущих манёвров.
- В совмещённом режиме объединяет статическую устойчивость Game Theory с динамической достижимостью Complexity Theory.
- Показывает live progress: реальные завершённые этапы, текущий этап, таймер анализа, число профилей и счётчик вида `4 из 16 проф.`.
- Стримит ответ LLM по фазам, ставит запросы в очередь и позволяет повторить проблемный запрос без потери уже рассчитанных этапов.
- Формирует развёрнутую статью агента на русском языке в формате объяснения механики игры для менеджера продукта.
- Собирает план действий для PM: что делать дальше, почему сейчас, стратегические ходы, план проверки, ограничители запуска и сценарии контрходов.
- Визуализирует акторов, стратегии, выигрыши, попарные срезы и ветвление стратегий через Recharts.
- Хранит историю анализов, сортирует кейсы по дате создания от новых к старым и показывает статус, длительность, число профилей и число фрагментов ответа.
- Экспортирует полный анализ в печатный PDF-документ: белый фон, шрифт 10 pt, минимальные рамки, без пустых страниц в конце.
- Позволяет выбрать LLM-провайдера в разделе `Настройки`: локальная LLM через LM Studio или облачная Yandex AI Studio.

## Архитектура

Проект — full-stack TypeScript-приложение.

### Клиент

- React + Wouter.
- Tailwind CSS и shadcn/ui.
- TanStack Query для API-запросов.
- Recharts для графиков и Sankey-like представлений.
- Основные страницы:
  - `NewAnalysis.tsx` — создание кейса, выбор режима анализа и live progress.
  - `AnalysisView.tsx` — результат анализа, визуализации, статья агента, PM-пакет и ссылка на PDF.
  - `History.tsx` — список кейсов, статусы, метрики, длительность и сортировка по дате создания.
  - `Settings.tsx` — выбор LLM-провайдера и параметры подключения.

### Сервер

- Express API.
- SSE-стриминг прогресса анализа.
- Очередь запросов к LLM для последовательного выполнения фаз.
- OpenAI-compatible интеграции для LM Studio и Yandex AI Studio.
- Обработка ошибки LM Studio `Model reloaded`: анализ не считается безвозвратно упавшим, пользователь может проверить загруженную модель и повторить тот же запрос.
- Проверка русскоязычности человекочитаемых полей JSON от LLM.
- Генерация PDF через `pdfkit`.
- Локальный debug-режим без реальной LLM для проверки продукта и UI.

### Общие типы и данные

- `shared/schema.ts` — Drizzle-схема и общие TypeScript-типы.
- `server/db.ts` — SQLite + Drizzle.
- `server/complexity.ts` — промпты, JSON-контракт, нормализация и bounded adaptive simulation для Complexity Theory.
- `server/complexity.fixtures.ts` — smoke-fixture для проверки контракта режима Complexity Theory.
- `script/build.ts` — production-сборка клиента и серверного bundle.

## Поток Game Theory

1. Пользователь описывает кейс: что запускаем, для какого сегмента, почему сейчас, какую ценность ждём и какие реакции участников беспокоят.
2. Агент выделяет игроков, границы игры, стратегии и допущения.
3. Бэкенд ограничивает модель до управляемого числа игроков и профилей.
4. Стратегические профили оцениваются пакетами, по умолчанию по 4 профиля за один LLM-запрос.
5. Сервер объединяет оценки, рассчитывает равновесия, Nash score, confidence и сводные метрики.
6. LLM генерирует развёрнутую статью агента на русском языке.
7. LLM собирает план действий для продакт-менеджера.
8. Результат сохраняется в истории и становится доступен как интерактивная страница и PDF-отчёт.

## Поток Complexity Theory

1. Пользователь выбирает режим `Complexity Theory` и описывает кейс обычным текстом.
2. LLM собирает адаптивных игроков, переменные состояния, обратные связи, пороги, зависимости от ранних событий и возможные вмешательства.
3. Сервер прогоняет ограниченную адаптивную симуляцию: базовый сценарий, ускоренный рост и стресс-сценарий, каждый по 8 шагов.
4. Сервер выявляет режимы системы, ранние сигналы, триггеры смены режима и первичные метрики устойчивости.
5. LLM формирует развёрнутый русский анализ и управленческие рекомендации без логики равновесия Нэша.
6. Результат сохраняется в истории, показывается отдельным дашбордом и доступен в PDF.

MVP сознательно не является полноценной agent-based simulation. Он не моделирует тысячи агентов и не калибрует вероятности на исторических данных. Цель режима — быстро и объяснимо показать PM нелинейные риски запуска до передачи задачи в разработку.

## Поток совмещённого анализа

Совмещённый анализ использует гибридную рамку Arthur/Sandholm:

1. Строит Game Theory-модель и определяет целевое устойчивое состояние.
2. Строит Complexity Theory-модель и проверяет, как система может двигаться к этому состоянию после запуска.
3. Сравнивает целевое равновесие с динамическими траекториями.
4. Показывает, где устойчивое состояние достижимо, где оно ломается, какие ранние сигналы важны и какие вмешательства нужны.
5. Собирает итоговый PM-пакет: запускать, ограничивать, отложить или менять механику до разработки.

## LLM-провайдеры

Провайдер выбирается на странице `Настройки`.

### LM Studio

Подходит для приватной локальной работы с чувствительными продуктовыми кейсами.

```env
LLM_PROVIDER=local
OPENAI_BASE_URL=http://127.0.0.1:1234/v1
OPENAI_API_KEY=lm-studio
LLM_MODEL=auto
LLM_TIMEOUT_MS=900000
LM_STUDIO_TTL_SECONDS=86400
```

### Yandex AI Studio

Подходит для облачного режима, когда нужна более стабильная или производительная модель. API-ключ не хранится в README и не должен попадать в git. Пользователь вводит его на странице `Настройки` или задаёт через локальный `.env`.

```env
LLM_PROVIDER=yandex
YANDEX_AI_STUDIO_BASE_URL=https://ai.api.cloud.yandex.net/v1
YANDEX_AI_STUDIO_API_KEY=
YANDEX_AI_STUDIO_PROJECT_ID=<your-project-id>
YANDEX_AI_STUDIO_MODEL=<your-model-uri>
YANDEX_AI_STUDIO_TEMPERATURE=0.8
YANDEX_AI_STUDIO_MAX_OUTPUT_TOKENS=40000
YANDEX_AI_STUDIO_TIMEOUT_MS=900000
```

Локальный файл `.nash-agent-settings.json` используется для пользовательских настроек и исключён из git.

## Русский язык в результатах

JSON от LLM валидируется на человекочитаемые поля. Если модель возвращает английский текст в результате, анализ завершается понятной ошибкой с просьбой переписать текстовые значения на русском языке. Это защищает UI и PDF от смешения языков.

## PDF-отчёт

На странице анализа в самом низу есть ссылка на скачивание PDF:

```text
/api/analyses/:id/pdf
```

PDF включает все разделы анализа, доступные для выбранного режима:

- входные данные и состав модели;
- игроки, стратегии, допущения и сводные участники;
- метрики, равновесия, выигрыши, риски и ограничения;
- визуализации и попарные срезы;
- развёрнутая статья агента;
- план действий для продакт-менеджера;
- сценарии, режимы, ранние сигналы и триггеры для Complexity Theory и совмещённого анализа.

## Локальный запуск

Установите зависимости:

```bash
npm install
```

Создайте `.env` на основе `.env.example`:

```bash
cp .env.example .env
```

Запустите приложение:

```bash
npm run dev
```

По умолчанию сервис поднимается на:

```text
http://127.0.0.1:5000/#/
```

Порт можно изменить через `PORT`. В локальной разработке удобно использовать `PORT=5001`, если `5000` уже занят.

## Проверка

TypeScript:

```bash
npm run check
```

Production-сборка клиента и сервера:

```bash
npm run build
```

Проверка PDF после изменений в генераторе отчётов:

```bash
curl -L http://127.0.0.1:5001/api/analyses/45/pdf -o /tmp/nash-analysis.pdf
pdfinfo /tmp/nash-analysis.pdf
```

## Не коммитить

- `.env` — локальные переменные окружения.
- `.nash-agent-settings.json` — пользовательские настройки LLM и сохранённые ключи.
- `nash.db` — локальная SQLite-база с анализами.
- API-ключи, токены и приватные параметры подключения.

Не добавляйте секреты в README, `.env.example`, исходный код или историю git.
